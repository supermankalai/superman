Episode 2: Battle with Backpressure

How I tuned Flink checkpoints, buffers, and parallelism to keep the stream flowing under heavy load.

Understanding Backpressure in Flink

Backpressure occurs when data producers generate messages faster than consumers can process them. In Flink, this manifests as:

  • Increasing buffer usage
  • Slowing processing rates
  • Checkpoint failures
  • Eventually, job failures if not addressed

Backpressure Indicators

  • High buffer usage (>70% utilization)
  • Increasing checkpoint durations
  • Frequent backpressure status in Flink UI
  • Growing consumer lag in source connectors
  • CPU saturation on task managers

Detecting Backpressure

Flink provides multiple ways to detect backpressure:

Flink Web UI Backpressure Monitoring

The Flink UI shows backpressure status for each task:

  • OK: No backpressure
  • LOW: Mild backpressure
  • HIGH: Severe backpressure

Metrics-Based Detection

Monitor these key metrics:

# Key backpressure metrics
flink_taskmanager_job_task_buffers_inPoolUsage
flink_taskmanager_job_task_buffers_outPoolUsage
flink_taskmanager_job_task_buffers_inQueueLength
flink_taskmanager_job_task_buffers_outQueueLength

Common Causes of Backpressure

Identify the root cause before applying fixes:

Source-Related Issues

  • Kafka consumer跟不上生产速率
  • Network latency in source connectors
  • Serialization/deserialization bottlenecks
  • Insufficient source parallelism

Processing Bottlenecks

  • CPU-intensive operations
  • I/O-bound operations (database calls, external APIs)
  • State operations (RocksDB performance)
  • Window operations with large state
  • Complex event processing patterns

Sink-Related Issues

  • Slow external systems (databases, APIs)
  • Network latency to sink systems
  • Insufficient sink parallelism
  • Batch size configuration issues

Strategies to Mitigate Backpressure

1. Increase Parallelism

Scale out your Flink job by increasing parallelism:

// Set parallelism in Flink configuration
StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment();
env.setParallelism(16); // Increase from default 1

// Or set per operator
dataStream
    .map(new MyMapper())
    .setParallelism(8);

2. Optimize Checkpointing

Configure checkpoints to minimize impact:

CheckpointConfig config = env.getCheckpointConfig();
config.setCheckpointInterval(60000); // 60 seconds
config.setCheckpointTimeout(300000); // 5 minutes
config.setMinPauseBetweenCheckpoints(30000); // 30 seconds
config.setTolerableCheckpointFailureNumber(3);

3. Buffer and Batch Optimization

Adjust network and memory buffers:

# Flink configuration for buffers
taskmanager.memory.segment-size: 32kb
taskmanager.network.memory.buffers-per-channel: 2
taskmanager.network.memory.floating-buffers-per-gate: 8

4. State Backend Optimization

For RocksDB state backend:

# RocksDB optimization
state.backend.rocksdb.thread.num: 4
state.backend.rocksdb.writebuffer.size: 64mb
state.backend.rocksdb.block.cache-size: 256mb
state.backend.rocksdb.writebuffer.count: 4

Advanced Techniques

Dynamic Scaling with Reactive Mode

Flink's reactive mode automatically scales based on backpressure:

# Enable reactive mode
jobmanager.scheduler: adaptive
jobmanager.adaptive-scheduler.resource-wait-timeout: 5min
jobmanager.adaptive-scheduler.min-parallelism: 4
jobmanager.adaptive-scheduler.max-parallelism: 64

Backpressure-Aware Routing

Implement custom routing based on downstream pressure:

public class BackpressureAwareRouter implements ProcessFunction {
    @Override
    public void processElement(Event event, Context ctx, Collector out) {
        if (isDownstreamBackpressured()) {
            // Route to alternative path or buffer
            bufferEvent(event);
        } else {
            out.collect(event);
        }
    }
}

Rate Limiting and Throttling

Control input rates to prevent overload:

// Implement rate limiting
dataStream
    .assignTimestampsAndWatermarks(
        WatermarkStrategy
            .forBoundedOutOfOrderness(Duration.ofSeconds(5))
            .withIdleness(Duration.ofMinutes(1))
    )
    .process(new RateLimitingProcessFunction(1000)); // 1000 events/sec

Monitoring and Alerting

Set up comprehensive monitoring:

Key Metrics to Alert On

  • Backpressure ratio > 0.5 for more than 5 minutes
  • Checkpoint duration > checkpoint interval
  • Buffer usage > 80% for extended periods
  • Consumer lag increasing continuously
  • Task manager CPU > 90% utilization

Case Study: Real-World Backpressure Resolution

In a recent project, we faced severe backpressure in a sessionization job:

  • Problem: 15-minute checkpoints, HIGH backpressure
  • Root Cause: RocksDB state operations + window aggregation
  • Solution: Increased parallelism from 8 to 24, optimized RocksDB config, implemented incremental checkpointing
  • Result: Checkpoints reduced to 45 seconds, backpressure eliminated

Proactive Backpressure Prevention

Build backpressure resilience into your architecture:

  • Design for horizontal scaling from day one
  • Implement circuit breakers for external calls
  • Use async I/O for database operations
  • Monitor proactively, not reactively
  • Test under load during development
