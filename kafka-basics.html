Episode 1: The Kafka Awakens

Mastering Apache Kafka fundamentals: Architecture, topics, partitions, and making brokers happy when the firehose starts.

What is Apache Kafka?

Apache Kafka is a distributed streaming platform that enables you to build real-time data pipelines and streaming applications. It's designed to handle high volumes of data with low latency, making it perfect for modern data-intensive applications.

Key Kafka Concepts

  • Topic: A category or feed name to which messages are published
  • Partition: Topics are split into partitions for parallelism
  • Producer: Applications that publish messages to topics
  • Consumer: Applications that subscribe to topics and process messages
  • Broker: Kafka server that stores data and serves clients
  • Cluster: Multiple brokers working together

Kafka Architecture Deep Dive

Kafka's architecture is built for scalability and fault tolerance. Here's how it works:

Topics and Partitions

Topics are divided into partitions, which are ordered, immutable sequences of messages. Each partition is replicated across multiple brokers for fault tolerance.

# Example topic creation with partitions
kafka-topics.sh --create \
  --topic orders \
  --partitions 6 \
  --replication-factor 3 \
  --bootstrap-server localhost:9092

Best Practice: Partitioning Strategy

Choose the number of partitions based on:

  • Expected throughput (more partitions = more parallelism)
  • Consumer processing time
  • Cluster size and resources
  • Start with 6-12 partitions and scale as needed

Producers: Publishing Messages

Producers are responsible for publishing messages to Kafka topics. Key configuration options:

# Producer configuration example
Properties props = new Properties();
props.put("bootstrap.servers", "localhost:9092");
props.put("key.serializer", "org.apache.kafka.common.serialization.StringSerializer");
props.put("value.serializer", "org.apache.kafka.common.serialization.StringSerializer");
props.put("acks", "all"); // Ensure message durability
props.put("retries", 3); // Retry on failure
props.put("compression.type", "snappy"); // Reduce network traffic

Message Keys and Partitioning

Messages can have keys that determine which partition they go to. Same key = same partition (ordering guarantee).

Warning: Message Size Limits

Kafka has default message size limits (1MB). For larger messages:

  • Configure message.max.bytes on brokers
  • Set max.request.size on producers
  • Consider compression for large payloads
  • Alternatively, store large data in external storage and send references

Consumers: Reading Messages

Consumers read messages from topics and process them. They work in consumer groups for parallel processing.

# Consumer configuration example
Properties props = new Properties();
props.put("bootstrap.servers", "localhost:9092");
props.put("group.id", "order-processors");
props.put("key.deserializer", "org.apache.kafka.common.serialization.StringDeserializer");
props.put("value.deserializer", "org.apache.kafka.common.serialization.StringDeserializer");
props.put("auto.offset.reset", "earliest"); // Start from beginning if no offset
props.put("enable.auto.commit", "false"); // Manual offset commit for exactly-once

Consumer Groups and Rebalancing

Consumer groups allow parallel processing of topics. Each partition is consumed by exactly one consumer in the group.

Exactly-Once Semantics

Kafka supports exactly-once processing through transactional producers and consumer offset management:

# Enable exactly-once semantics
props.put("enable.idempotence", "true");
props.put("transactional.id", "unique-producer-id");

// In producer code
producer.initTransactions();
try {
    producer.beginTransaction();
    producer.send(record1);
    producer.send(record2);
    producer.commitTransaction();
} catch (Exception e) {
    producer.abortTransaction();
}

Performance Optimization

Optimize Kafka for high throughput and low latency:

  • Batch Size: Larger batches improve throughput but increase latency
  • Linger.ms: Wait time for batching messages
  • Compression: Use snappy or lz4 for better network utilization
  • Acks: Trade-off between durability and performance
  • Replication: Ensure adequate replication factor for durability

Monitoring and Observability

Essential metrics to monitor:

  • Producer/Consumer throughput and latency
  • Broker CPU, memory, and disk usage
  • Consumer lag (messages behind)
  • Network throughput
  • Topic partition sizes and growth rates

Common Pitfalls and Solutions

Avoid these common Kafka mistakes:

  • Too many partitions: Can cause overhead and rebalancing issues
  • Improper key selection: Leads to data skew and hot partitions
  • Ignoring consumer lag: Can cause data processing delays
  • Insufficient replication: Risk of data loss during broker failures
  • Poor retention policies: Either too short (data loss) or too long (storage costs)
